我的工作时间安排反思

## 问题背景

工作的当中的时间安排是非常重要的，合理的安排自己花在一件事情上的时间可以尽可能的增加你的投入产出比。在相等的产出要求的情况下，越高的产出比可以尽可能的减少你付出的时间，这个是非常重要的。工作的很大部分其实都只是完成工作，工作当中对自身提升有利的部分其实占的比重很少，而且随着经验的累积和对技术的熟练，这些平时的业务开发任务越来越不能提升自身的技术积累。所以我们需要尽可能地减少花在这些琐碎的业务开发中的时间，把剩余的时间花在自我的学习提升上。怎么花时间到自我提升上，要提升什么又是另外一个话题，之后再说。那么怎么减少花在业务开发中的时间呢？其实我们知道：时间=总量/效率，要不就减少总量，要不就提高效率。

所以大的方向可以分为如下两个部分：

* 减少这些业务开发的活儿
* 提升业务开发的效率

## 减少业务开发量

其实减少业务开发量是最直接的减少所花时间的方法，不仅如此，减少业务开发量其实还可以提升你的业务开发的效率，这个逻辑稍后再讲。但是很少有人可能会关注这个问题。很多人可能有这样的误解，觉得对于开发工作人员来说就是去完成产品提出的需求，老板分配的任务，没有自己去减少业务开发量的权利，其实这是很大的错误。最起码我现在总结到有如下的方式去减少你的工作量：

1. 合理地质疑需求

产品同学提出的需求不一定好，我总结了一下产品同学提出的需求经常会有以下的问题：

* 站在产品的角度提出的方案会受到技术的限制，有的时候并不能提出一个即有利于开发实现，又能满足产品需要的方案。或者说命名有一种方法即好做，又能达到一样的效果，但是产品却非选择了一种不好实现的。
* 由于产品同学只是在产品未实现之前脑补的整个产品逻辑，难免会有细节考虑不全的地方，这种情况经常是需要开发同学在开发的过程中发现并弥补的，这种时候就需要开发同学提出来然后给一个合理的解决方式
* 产品在提出需求的时候可能过于复杂，本来是没必要的，没有考虑用户的学习成本导致需求过于复杂从而增加开发工作量
* 产品在提出需求的时候没有考虑一个功能的投入产出比，其实有的时候一些不太重要的细节功能对于产品来说可能只是一个简单的按钮，感觉很简单，但是从开发的角度来看反而要花很多的时间。尤其在项目的迭代任务比较重的时候，应当先满足主要的重点功能，这些细节的优化功能都是可以和产品同学argue并把任务后移或者砍掉的。这一点尤为重要。

所以，在遇到产品的需求的时候，要审视一下是否有如上的问题，一定要站在产品的角度去理解这个产品，要明白这个功能解决了什么问题。从开发的角度去给出一些即好实现又能满足需求的方案。在项目周期紧张的时候，要学会把握功能的主干，砍掉或者后移投入产出比不高的功能。

2. 合理的拒绝需求

* 当一个需求在评审的时候，如果你的时间不够，排期已经满了就要大胆的拒绝需求，如果你接了反而做不完对谁都没好处
* 如果一个问题比如一个bug本来就不是你的问题，那么就一定不要轻易的让步去让你去兼容。是后端的问题就要让后端去解决，如果事产品的问题就要让他新需求来走排期。就算是要前端参与也要充分的了解为什么要我来改，是不是非得让我来改？如果是的话才需要做这个事情，而且还需要充分的评估做这个事情的难易程度，如果花的时间太多就算要让我来修复，那也得往后移。这些本来就不是你的锅导致的问题都是可以拒绝的。

3. 给项目的时间留下充足的buffer

* 一定不要为了面子把项目的时间评估的很紧张，要留出充足的buffer

对于减少自己的业务开发量，那么我们就可以从以上三点着手，合理的对待我们的业务开发，减少用在业务开发上的时间。有的同学可能就会疑惑了，减少自己的业务开发量是不是意味着对工作不负责任，绩效不好看？实际上决定你绩效的从来不是这些搬砖的工作，你搬了再多的砖也不能让你的绩效变好，更别提绩效本来也是一种管理员工的手段，也别提搬砖本来就是为资本打工，我们要对自己的工作负责，但是不能负责过头，我认为的负责就是在力所能及的工作时间里面把你自己认领分配到的活儿保质保量的完成即可。如果你想要提升你的绩效绝不是搬了更多的普通的砖，而是在搬砖之余为团队的发展做出了更积极的贡献，或者你运气好进了一个好的项目，但这种运气不是个人能掌控的，这点就不展开细说了。

## 提高开发效率

除了减少开发任务外，我们还需要合理地分配自己的时间，从而最高效地完成自己的开发任务。

为了理清楚完成任务的方式，我们首先应该对这个问题建立一个合理的模型。人的大脑其实就像是一个cpu,不同的任务就是不同的进程。为了专注提高cpu的产出，我们干活的最佳方式就是在一个时间下专注一件事情，去执行一个进程中的线程。首先我们考虑一个理想的情况那就是所有的事情都是同一个优先级，收益都一样，做一分钟的事情就得到一分的收获，并且不存在相互依赖的关系，那么我们只需要随机的选取要进行的任务，一个接一个的完成即可。在这种情况下，唯一限制我们工作的产出的就是工作的时间，这种情况下我们只需要保持健康，安心工作即可。

但是在实际的工作情况下，工作的内容不可能是没有优先级的，不同的任务有相互依赖的关系，还有因为协作的原因阻塞你的工作进度，还有可能突然飞来一个别的任务要打断你的当前进度等等等等。人的大脑和cpu一样，要处理一样东西的时候要先把相关的信息放在内存里，然后再进行运算，如果被打断的话，我们的大脑就会把当前的内容清空，换成另外一个东西到内存中进行运算。我们都知道这样的I/O操作是相当花时间的。所以我们应该在尽可能完成优先级高的任务的情况下，尽可能少的切换任务。

从上面的论述中，我们可以理解到，提高开发的效率无外乎两种方式：

* 优先完成优先级高的任务
* 尽可能少的减少切换任务带来的I/O成本

当然提高熟练度什么的也是提高开发效率的一部分，但是并不在这篇文章的讨论范围内。

接下来就围绕这两个部分的内容来进行探讨如果提高开发效率：

### 把时间花在优先级更高的事情上

一件事情如何评判优先级？我觉得首先要引入一个delay的概念。什么是delay呢？当两个任务有依赖关系的时候，一个任务必须等待另外一个任务执行完才能执行，这个任务等待的时间就是delay。对项目管理来说，我们就是要让所有的任务的总的delay时间最小化。那些可以让全局delay减少最多的事情就是优先级最高的事情。对自己来说也是一样，所以我们来看一下先做哪一些事情可以让我们的delay减少的最多：

对于我现在阶段的项目开发人员来说，优先级更高的事情可以总结为一下几点：

* p0测试用例要求的功能点

  * p0测试用例十分重要，是从测试测试关系的角度来进行的优先级划分，如果不过的话将阻碍整个测试进度导致整体delay增加，而且还会因为测试时间倒逼导致情绪金正效率下降
* 需要等待他人参与的功能点

  * 之所以需要优先完成等待他人参与的功能点，是为了让等待的delay时间是避免不了的，我们需要在这个delay的事情上填充不会被delay的事情。整体的进度才不会被拖累。比如在联调的过程当中，去验证后端接口的合理性比前端一些界面效果的展示更加的重要。不仅因为后端参与的功能点一般是p0相关，还因为在给后端提出了bug过后，后端在修改的时间是需要让前端等待的。这个等待的时间需要被留给不需要他人参与的功能点填充避免无效的等待。
* 领导安排的重要的事情

  * 原因就不说了
* 时间deadLine快到了的事情

  * 如果时间快到了那就别提什么delay和依赖的关系了，赶紧做吧。但是这种事情应该尽可能的避免。如果因为拖延导致这种事情变多了自然全局的delay也就多了。
* 今天一定要完成的事情

  * 有一些事情比如合流这种一定要在今天完成的事情，那就应该把优先级放到最高

### 尽可能地减少切换任务带来的I/O成本

尽可能减少I/O成本的原因在刚才也论述的很清楚了，那么怎么减少I/O成本呢？我认为有两个方面：

1. 减少切换的次数
2. 减少切换的时候的I/O量

第一个方面应该比较好理解，原因就不过多阐述了。那么有哪些方式可以实现第一点呢？我认为有如下的一些技巧：

* 以一个番茄时间25分钟为一个周期，在这个周期中尽可能的做一件事情，在这个番茄时间中遇到的打断任务都放在收集箱中，等一个番茄时间过了再去统一处理
* 在完成一个任务的时候不要顺手迭代过深入的内容。比如你本来想要完成功能A，结果发现完成功能A的时候发现了一个无关的bug,又顺手又修复了一个bug,这个bug下面又有一个问题要解决。。。。这样就会发现你离要解决的任务A越来越远。

第二个方面是什么意思呢，当有一些任务是没有办法避免切换的时候，那么我们应该尽可能地保留当前的开发信息，让我们再一次切换回来的时候能够直接就回忆起我们在干啥，干到哪里了，不要再来回找很多东西。这一点也有很多技巧可以使用：

* 进行项目任务的分解，在腾讯文档上进行的文档分解可以帮助你保留项目的总体进展，方便定位where you are
* 项目任务中的剩余问题栏目可以帮助你很快的切换到剩余的问题中
* 在进行的任务的滴答清单中记录你现在的工作进展在哪里，方便回溯

## 项目开发流程图

![](/Users/zhouxihang/Desktop/my-projects/docs/docs/assets/2022-01-09-19-46-03-image.png)

之前都是关于整个方法论的理论分析，但是除了理论的分析，实践也同样重要。以上是整个工作流程的逻辑图。接下来是对每个模块进行细节地详细补充说明：

1. 进行需求拆解到腾讯文档中

这个步骤非常重要，但是不需要拆分得太细，只拆分到功能的粒度即可

2. 每天上班的时候依据文档中的项目总览和清单中的任务提醒根据优先级进行今天任务的分配

这个步骤在生成滴答清单中的任务的时候，要注意任务的粒度不能大于功能的粒度，否则就会造成一个任务涵盖了太多的内容，里面的细节就会过多

3. 对不重要的事情加入收集箱

这里不重要的事情可以是别人来找你的事情，也可以是你自己在写代码的过程中发现的别的问题。遇到这种问题都统一加入收集箱中。不要打断当前的进度

4. 需要等待时切换到第二任务

需要明确两个问题：什么是需要等待的时机，什么是第二任务。

需要等待的时机：当前执行的任务执行不下去了，比如需求有一点不明确了需要去问产品，问了过后在等待回复，比如后端联调出bug了要等待修复。

第二任务：优先级第二的任务，但是由于任务划分粒度的原因，不可能每个滴答清单上的任务都是不可在划分的任务。即有可能你当前在执行的任务虽然某一个方向被阻塞了，但是当前的任务还是有其他的任务可以做，那就去做这些事情。

这种需要等待的任务需要在当前的任务栏上标⌛️。注意⌛️符号是因为需要等待他人而画的符号，不意味着等等在做。

5. 打断任务需要马上执行

这种情况在25分钟的跨度里面很少发生，真的发生了也就新建一个任务，然后保留当前任务的进度即可。在一个任务中划分了小任务后表示当前正在做的任务使用🏃‍♀️。

6. 番茄时间内完成任务

如果在一个番茄时间内完成了任务，或者是由于当前任务被拆分过后剩下的部分优先级不够了，就把剩下的部分加入到文档中对应的位置的剩余问题中，然后进行下一个优先级任务。

7. 保存当前工作状态

如果休息时间到了当前的任务还没有完成，就要在清单中检查一下，使用🏃‍♀️保存一下当前的工作状态。

8. 处理收集箱中的任务

* 把对应的任务进行归类

  * 首先是归类到对应的滴答清单的项目中
  * 如果是项目相关的话要加入到文档对应位置的剩余问题中❕
  * 于此同时需要确定这些问题的完成时间
* 如果是一些比较小的任务那么就可以马上完成，比如说5分钟以内的，但是这种事情比较少，一般有的话也是生活中的一些琐事。

9. 处理邮件和微信中的信息

* 邮件中的琐事同样需要进行归档
* 回复企业微信中的消息
* 把需要完成的任务加入到文档中

10. 重新评估优先级

在进行下一个番茄之前需要重新评估一下当前工作的优先级

* 有可能被打断的高优先级工作可以恢复
* 有可能新出现的问题成为了最高优先级的事情

11. 在下班之前提醒自己要归档问题

* 没有解决的问题要归档到文档上
* 需要提醒自己的时间结点加入到清单上

| 表情     | 使用场景                                                                                         |  |
| -------- | ------------------------------------------------------------------------------------------------ | - |
| ⌛️     | 被他人阻塞导致任务不能进行需要等待                                                               |  |
| 🏃‍♀️ | 用在任务的说明中，表示任务在同级别任务中正在进行                                                 |  |
| ✅       | 用在任务的说明中，表示任务已经完成                                                               |  |
| ⌚️     | 用在任务的说明中，表示任务的优先级不够当前不执行，需要在其他的子任务完成后加入到文档的剩余问题中 |  |
| 🍉       | 用在任务中，表示在同级别任务中的优先级高，需要优先完成                                           |  |
